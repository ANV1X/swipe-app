# Swipe Fashion — Telegram Mini App

Свайп-приложение для подбора одежды (Tinder для шопинга). Полностью рабочий
стек: **React-фронтенд + собственный backend на FastAPI + база данных
PostgreSQL/SQLite**. Никакого Supabase — все данные хранятся в вашей базе,
все кнопки в интерфейсе подключены к реальным API-запросам.

## Структура репозитория

```
swipe-app/
├── backend/     — FastAPI + SQLAlchemy + PostgreSQL (или SQLite для разработки)
└── frontend/    — React + Vite (работает и как Telegram Mini App, и как обычный сайт)
```

## Что умеет backend

Полноценный REST API на FastAPI, покрывающий весь функционал приложения:

| Раздел | Возможности |
|---|---|
| Товары | лента с фильтрами по категории/цене/полу/**стилю/цвету**, скидки, **персонализация по вкусу и истории лайков**, история цены |
| Свайпы | лайк/дизлайк (уже просмотренное не повторяется в ленте), автодобавление в вишлист, история |
| Вишлист | добавление/удаление, уведомления о снижении цены |
| Коллекции | **создаются любым пользователем**, официальные (от админов) и пользовательские — отдельные вкладки, можно добавлять/убирать товары, делиться с друзьями |
| Батлы | **соревнуются коллекции пользователей**, а не случайные товары — выставляете свою коллекцию в очередь, система находит соперника; у батла есть приз |
| Друзья | **реальные пользователи приложения** — связь устанавливается по инвайт-ссылке/коду, не выдуманные контакты |
| Шеринг | можно поделиться товаром или коллекцией с другом — ему придёт уведомление |
| Совместные вишлисты | создание, приглашение по ссылке, добавление товаров |
| Уведомления | автогенерация при снижении цены, подписке на коллекцию, добавлении друга, шеринге, старте батла |
| Профиль | реальная статистика, достижения, кэшируется на фронте — открывается мгновенно |
| Рефералы | свой код/ссылка, по переходу — и реферал, и реальная дружба одновременно |
| Админка | **доступна только администратору** (см. ниже), дашборд, экспорт CSV/JSON, создание батлов/коллекций |

Авторизация работает в двух режимах одновременно:
- **Telegram Mini App** — через подпись `initData` (проверяется по `BOT_TOKEN`);
- **Обычный браузер** (например, превью на Vercel вне Telegram) — через
  анонимный идентификатор, который фронтенд сам генерирует и сохраняет в
  `localStorage`.

Это значит, что приложение полностью работает и тестируется даже без
настройки Telegram-бота.

## Кому доступна админка

Админка (`/admin`) больше **не открыта всем по умолчанию**. Доступ только у
пользователей, чей `telegram_id` указан в переменной окружения `ADMIN_IDS`
на backend (через запятую, если админов несколько).

1. Узнайте свой `telegram_id` — например, напишите боту `@userinfobot` в Telegram.
2. На Railway откройте сервис backend → **Variables** → добавьте/измените `ADMIN_IDS=ваш_telegram_id`.
3. Передеплойте backend.

Без Telegram (анонимный браузерный режим) администратором стать нельзя — это
осознанное ограничение безопасности, поскольку у анонимных пользователей нет
проверяемой личности.

## Быстрый старт (локально)

### 1. Backend

```bash
cd backend
python3 -m venv venv && source venv/bin/activate   # опционально
pip install -r requirements.txt
cp .env.example .env      # по умолчанию использует локальный SQLite — ничего менять не нужно
uvicorn app.main:app --reload --port 8000
```

При первом запуске backend сам создаст таблицы и заполнит базу демо-товарами,
коллекциями и батлом (см. `app/seed.py`). Открыть документацию API:
http://localhost:8000/docs

### 2. Frontend

```bash
cd frontend
npm install
cp .env.example .env      # VITE_API_URL=http://localhost:8000 (уже стоит по умолчанию)
npm run dev
```

Откройте http://localhost:5173 — приложение полностью рабочее в обычном
браузере (без Telegram), так как использует анонимную авторизацию.

## Деплой в продакшен

### Backend → Railway / Render / любой хостинг с Python

> У вас уже есть Railway-проект на `honest-reflection-production-3f73.up.railway.app` —
> просто замените содержимое папки `backend/` в репозитории, который подключён
> к этому проекту, и запушьте. Procfile/railway.json уже настроены на нужную
> команду запуска, менять ничего не нужно.

1. Задайте переменную окружения `DATABASE_URL` (PostgreSQL) — например, Railway
   сам создаёт её при подключении Postgres-плагина.
2. (Опционально) `BOT_TOKEN` — если хотите авторизацию через Telegram.
3. (Опционально) `ADMIN_IDS` — telegram_id через запятую, кто видит `/admin`.
4. `CORS_ORIGINS` — домен фронтенда (например, `https://your-app.vercel.app`).
5. Деплой уже настроен через `Procfile` / `railway.json`
   (`uvicorn app.main:app --host 0.0.0.0 --port $PORT`).

⚠️ В вашей базе на Railway, скорее всего, уже есть таблицы по старой схеме.
Новый backend создаёт свои таблицы автоматически при старте, но если старые
таблицы с такими же именами уже существуют в несовместимом формате — будет
конфликт. Проще всего удалить и заново создать Postgres-плагин в Railway
(или вручную дропнуть старые таблицы) перед первым деплоем новой версии.

> **Важно:** в этой версии снова изменилась схема БД (товары получили поля
> стиля/цвета, батлы теперь между коллекциями, а не товарами, друзья — это
> реальные пользователи, а не текстовые контакты). Перед деплоем выполните
> в Railway → Postgres → Query:
> ```sql
> DROP SCHEMA public CASCADE;
> CREATE SCHEMA public;
> ```
> Иначе старые таблицы конфликтуют с новой схемой и backend упадёт при первом запросе.

**Если вы уже разворачивали более раннюю версию этого backend** — перед
деплоем текущей версии обязательно очистите базу (см. раздел «Если меняется
схема базы данных» ниже): в этой версии полностью переработаны батлы (теперь
это коллекции, а не товары), друзья (теперь это реальные пользователи, а не
произвольные записи) и добавлены новые поля у товаров и уведомлений.

## Если меняется схема базы данных

Каждый раз, когда в очередном обновлении появляются новые таблицы или поля
у существующих, `Base.metadata.create_all()` создаёт только то, чего ещё нет
— уже существующие таблицы он не трогает и не меняет. Если структура
изменилась существенно (как в этом обновлении), проще всего обнулить базу:

```sql
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
```

Выполните это в Railway → Postgres → Query, затем перезапустите backend —
он создаст все таблицы заново и засеет демо-данными.

### Frontend → Vercel / Netlify
1. Задайте `VITE_API_URL` = адрес задеплоенного backend.
2. `npm run build` → папка `dist/`.

### Подключение к Telegram
1. Создайте бота через [@BotFather](https://t.me/BotFather), получите `BOT_TOKEN`.
2. Укажите `BOT_TOKEN` в backend.
3. В BotFather настройте Mini App, указав URL задеплоенного фронтенда.

## База данных

По умолчанию используется SQLite (файл `backend/swipe.db`) — отлично подходит
для разработки и демонстрации без какой-либо настройки. Для продакшена
просто укажите `DATABASE_URL` в формате PostgreSQL — код ничего менять не
надо, SQLAlchemy сам переключится. Таблицы создаются автоматически при
старте (`Base.metadata.create_all`).

## Архитектура авторизации

```
Запрос фронтенда
   │
   ├─ есть X-Init-Data (открыто в Telegram)? → проверяем HMAC-подпись по BOT_TOKEN
   │                                            → telegram_id как id пользователя
   │
   └─ нет → X-Anon-Id (сгенерированный uuid из localStorage)
                                                → uuid как id пользователя
```

Оба сценария ведут в одну и ту же таблицу `users` — пользователь,
открывший приложение в браузере, а затем в Telegram, просто получит два
разных профиля (это ожидаемое поведение для MVP без полноценного login).
